# 25-minio

Single-node [MinIO](https://min.io/) object store on a Longhorn PVC, in its own
`minio` namespace. Originally lived inside `24-meeting-bot` as the bot's S3
backend; promoted here so it is a standalone, reusable store.

## Bundles (deploy order via `dependsOn`)

1. `secrets/` — sealed `minio-s3` secret (the MinIO root credentials).
   Generated by `02-secrets/install.sh` from the template under
   `02-secrets/templates/25-minio/`.
2. `.` (this Helm bundle) — the MinIO Deployment, a ClusterIP Service
   (`api` 9000 / `console` 9001), a Job that creates the `meeting-bot` bucket,
   and a Traefik `IngressRoute` exposing the console at `minio.<zone>`. The
   `storage/` dir is a Kustomize overlay (applied alongside the chart via
   `kustomize.dir`) holding the `minio-data` PVC and its dedicated
   `longhorn-minio-storage` StorageClass — RWO, 2 Longhorn replicas with hard
   node anti-affinity, mirroring the `12-jellyfin` pattern.

In-cluster clients reach the S3 API at
`http://minio.minio.svc.cluster.local:9000`. The web console is at
`https://minio.<zone>` (login with the `minio-s3` credentials).

## Credentials

The `minio-s3` credentials **must match** the `meeting-bot-s3` secret in the
`meeting-bot` namespace — meeting-bot authenticates to this MinIO with those
same keys. Fill both bundle files with the same values before sealing.

## First deploy

1. Fill real MinIO credentials into
   `02-secrets/bundles/25-minio/minio/minio-s3.yaml` (copy of the template;
   git-ignored) — same values as `meeting-bot-s3`.
2. Run `02-secrets/install.sh` (talks to the cluster's Sealed Secrets
   controller — **you** run this, kubeseal required).
3. Commit + push. Fleet reconciles secret → MinIO.
