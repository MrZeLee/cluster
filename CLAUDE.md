# CLAUDE.md

Project-specific guidance for working in this repository.

## This is a GitOps repo — Fleet owns the cluster

This repository is the **source of truth** for the Kubernetes cluster. It is
synced to the cluster by **Rancher Fleet**: every directory (or subdirectory)
containing a `fleet.yaml` is a Fleet bundle, and Fleet continuously reconciles
the committed state into the cluster.

**The workflow is: edit files → commit → push. Fleet applies the change.**

### Do NOT run commands against the live cluster

There is **no need** (and you should not try) to apply changes yourself:

- ❌ `kubectl apply` / `kubectl kustomize` / `kubectl patch` / `kubectl create`
- ❌ `helm install` / `helm upgrade` / `helm template`
- ❌ `kustomize build`
- ❌ anything else that talks to or mutates the cluster

Applying or rendering manifests locally is unnecessary: Fleet does the
rendering (Helm + Kustomize) and the applying. Don't reach for `kubectl
kustomize` to "validate" a bundle either — a normal YAML read is enough, and a
bad bundle surfaces in Fleet, not in a local build.

Read-only inspection of the live cluster (e.g. `kubectl get` to debug) is fine
**only when the user explicitly asks** to look at running state. Default to
editing files, not querying the cluster.

### How changes reach the cluster

1. Edit the relevant files in this repo.
2. Commit and push (only when the user asks — see the global guidelines).
3. Fleet detects the new commit and reconciles the affected bundles.

### Authoring conventions

- Each app lives in a numbered top-level directory (e.g. `12-jellyfin`,
  `18-owncloud`); within it, numbered sub-bundles (`01-…`, `02-helpers`) order
  deployment.
- Bundles are either Helm (`helm:` block in `fleet.yaml`, chart under the
  bundle) or Kustomize (`kustomize.dir` pointing at a folder with a
  `kustomization.yaml`), and sometimes both.
- Cross-bundle ordering uses `dependsOn` in `fleet.yaml`, matched against a
  bundle's `labels`. Give each bundle a distinct `app:` label and depend on it.
- Ingress is Traefik `IngressRoute`; the public domain comes from the
  `cloudflare_zone` / `cloudflare_zone_hyphen` Helm values, injected per app via
  `helm.valuesFrom` → secret `helper-values-secret` in the `traefik` namespace
  (`key: values.yaml`). The TLS secret is `<cloudflare_zone_hyphen>-tls`. Copy an
  existing app's `ingress.yaml` (e.g. `15-activepieces`) as the template.

## Secrets (Sealed Secrets) — how this repo does it

Secrets are **never** committed in plaintext. The flow is non-obvious, so:

- **Plaintext templates** live in `02-secrets/templates/<NN-app>/<namespace>/<name>.yaml`.
  Two shapes are supported (the same file kinds `02-secrets/install.sh` keys off):
  - `kind: Secret` with `stringData:` → sealed and applied verbatim
    (e.g. `activepieces-secrets`, our `ocis-secrets`).
  - a values file (no `kind:`) → wrapped into a generic secret with the file
    under `key: values.yaml` (the `*-values-secret` / `helper-values-secret`
    pattern Helm charts read via `valuesFrom`).
- **Real values** are filled into `02-secrets/bundles/<NN-app>/<namespace>/<name>.yaml`,
  a copy of the template. Everything under `bundles/**` is git-ignored
  (each leaf dir has a `.gitignore` of `*` / `!.gitignore`) so real secrets
  never get committed.
- **`02-secrets/install.sh`** walks `bundles/**`, runs `kubeseal`, and writes the
  encrypted `seal-secret-<name>.yaml` + a generated `kustomization.yaml` into the
  **app's own** `secrets/` dir (e.g. `18-owncloud/secrets/`). `kubeseal` talks to
  the cluster's Sealed Secrets controller, so **the user runs this**, not you.
- Each app's `secrets/` dir keeps a committed `fleet.yaml` (a Kustomize bundle,
  `app: <name>-secrets` label); the `seal-secret-*.yaml` + `kustomization.yaml`
  are generated by `install.sh`. So when adding a new app secret you author: the
  `templates/...` plaintext, the `bundles/.../.gitignore`, and the app
  `secrets/fleet.yaml` — then the user runs `install.sh` to produce the sealed
  output. See `02-secrets/README.md`.
